## 1. 產品背景與問題

### 1.1 目前的痛點

現有 Grafana Dashboard 的監控系統有以下限制：

| 痛點 | 說明 |
|---|---|
| **數據孤島** | 工程師需要同時看 Dashboard、維運文件、歷史事件記錄，切換多個系統才能診斷問題 |
| **被動式監控** | 異常發生後才能看到警報，沒有趨勢預測能力 |
| **專業門檻高** | 理解 Prometheus metrics 需要熟悉 PromQL，基層工程師難以直接解讀 |
| **維運經驗流失** | 過往故障處理的知識存在少數人腦中，新人不易傳承 |
| **分析效率低** | 每次問題排查需要手動撈資料、比對歷史、做趨勢分析，耗時 30 分鐘以上 |

### 1.2 為什麼現在要做

- Grafana 已有完整監控覆蓋（Linux OS、DNS、VMware、Node Exporter）
- LLM 成本大幅下降，本地部署（Ollama）已可實用
- 維運文件數位化程度提高，RAG 技術成熟
- 團隊有 Python + Grafana 生態經驗

### 1.3 不做的代價（Cost of Inaction）

- 工程師每次排查平均多花 **30 分鐘**，一個月假設 20 次問題 = **浪费 10 小時/月**
- 緊急事件時依賴特定資深工程師，無法delegation
- 系統問題惡化後才發現，Mean Time to Detection（MTTD）高

## 2. 產品目標

### 2.1 核心價值主張

> **「讓任何工程師都能像資深 SRE 一樣，快速診斷系統問題」**

### 2.2 產品願景（Vision）

- **短期（Phase 1）：** AI 即時解讀監控數據，異常偵測與趨勢預測
- **中期（Phase 2）：** AI 結合內部維運知識，給出有根據的建議而非通用回答
- **長期：** 主動式系統健康評估，預測性維護

### 2.3 成功定義

| 維度 | Phase 1 成功標準 |
|---|---|
| 效能 | 使用者進入 Dashboard 頁面後，30 秒內顯示 AI 分析卡片 |
| 準確性 | 異常偵測準確率 > 85%（z-score > 2.5 的都是真異常）|
| 覆蓋範圍 | 支援 Linux OS、DNS Bind、Node Exporter、VMware 四種 Dashboard 自動生成卡片 |
| 可用性 | 無需打字，自動根據當前頁面生成可點擊的分析卡片 |

## 3. 功能範疇

### 3.1 Phase 1 功能（MVP）

| 功能 | 說明 | 使用時機 |
|---|---|---|
| **AI Insight Cards（即時解讀）** | 根據當前 Dashboard 動態生成可點擊的分析卡片 | 使用者進入 Grafana 頁面，自動看到該頁面的異常警示、趨勢預測、優化建議 |
| **異常偵測** | 自動標記偏離正常範圍的 metrics | 凌晨 3 點記憶體突增 |
| **趨勢預測** | 顯示未來 7 個時間點的趨勢 | 「磁碟空間什麼時候會滿？」 |
| **互動式圖表** | Plotly 圖表，可 hover、zoom、export | 點擊卡片後，深入分析特定區間 |
| **聊天介面** | iframe 內的對話式體驗 | 點擊卡片展開 AI 完整回覆 + 圖表 + 異常標註 |

### 3.2 Phase 2 功能（知識增強）

| 功能 | 說明 | 使用時機 |
|---|---|---|
| **維運文件攝取** | 上傳 PDF/MD/DOCX，建立內部知識庫 | 新進工程師查維運手冊 |
| **AI + 知識庫問答** | AI 分析時自動引用內部文件 | 「上次類似情況怎麼處理的？」 |
| **文件來源標註** | AI 回覆中標示引用了哪份文件 | 需要進一步查閱原始文件 |

### 3.3 明確不含（Out of Scope）

- 自動觸發的排程分析（維持請求觸發，減少多餘運算）
- 多租戶 / 權限管理
- 即時訊息推播（LINE / Slack）
- Metrics 的 CRUD 操作
- 跨多個 Prometheus 實例查詢

## 6. 技術方案說明

### 6.1 技術選型理由

| 技術 | 選用原因 | 替代方案 |
|---|---|---|
| **Python FastAPI** | 高效能、自動化 OpenAPI 文件、Python AI 生態豐富 | Node.js / Go |
| **Ollama（本地 LLM）** | 免費、無網路延遲、資料不外洩 | OpenAI（付費、有延遲）|
| **Plotly（圖表）** | JSON 驅動、互動性強、CDN 可直接使用 | Chart.js、ECharts |
| **PostgreSQL + pgvector（Phase 2）**| 一個資料庫搞定關聯數據 + 向量搜尋、部署簡單 | ChromaDB、Qdrant（額外架設）|
| **Docker Compose** | 一行指令啟動完整環境、易於交付 | Kubernetes（過度複雜）|

### 6.2 為什麼不用 MCP Server

一開始考慮使用 Docker 官方提供的 [Prometheus MCP Server](https://hub.docker.com/mcp/server/prometheus)，但評估後放棄：

| 面向 | MCP Server | 自建 PrometheusQuerier（最終選擇）|
|---|---|---|
| 查詢確定性 | AI 决定查什麼，可能不精準 | 工程師定義 PromQL，100% 確定 |
| 回應速度 | AI → MCP → Prometheus，多一次跳轉 | 直接 HTTP，ms 級 |
| 適用情境 | AI Agent 自主探索 | 固定監控流程，確定性分析 |

**結論：** 此系統是確定的監控流程，適合自建；未來若要做「AI 自主探索 Prometheus」的場景，再考慮 MCP。

## 7. AI 分析邏輯說明

### 7.1 即時解讀（AI Insight Cards）

**互動模式：卡片式引導，而非自由輸入**

使用者進入 Grafana 特定 Dashboard 頁面（如 Linux OS）時，iframe 內的 AI Chat 不需要使用者打字，而是自動呈現**一組動態生成的卡片**，每張卡片對應該頁面的關鍵洞察：

```
┌─────────────────────────────────────────────────────────────┐
│  💬 AI 監控分析                                             │
│                                                             │
│  基於你目前觀看的是「Linux OS Dashboard」，以下是建議的分析：  │
│                                                             │
│  ┌─────────────────┐ ┌─────────────────┐ ┌──────────────┐ │
│  │ ⚠️ 記憶體即將用盡  │ │ 📈 CPU 趨勢上升  │ │ 💡 優化建議   │ │
│  │  使用率達 87%    │ │  未來 3 小時可能  │ │ 可考慮重啟    │ │
│  │  [查看分析]      │ │  突破警戒線       │ │ log 服務      │ │
│  └─────────────────┘ └─────────────────┘ └──────────────┘ │
│                                                             │
│  ┌─────────────────┐ ┌─────────────────────────────────┐  │
│  │ 🔴 磁碟空間不足   │ │ 📊 異常事件摘要                  │  │
│  │  /dev/sda1 已用  │ │  過去 24 小時共偵測到 3 次異常   │  │
│  │  達 92%         │ │  [下載報告]                       │  │
│  └─────────────────┘ └─────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

**卡片如何生成：**

1. 系統根據目前 Grafana 頁面 URL，推斷是哪種 Dashboard（Linux OS / DNS / VMware / Node Exporter）
2. 向 Prometheus 查詢該 Dashboard 的核心 metrics（過去 24 小時數據）
3. AI 分析後，自動生成 3-6 張「情境感知卡片」：
   - **異常警示卡**（如「記憶體即將用盡」）— 該頁面有異常的 metrics
   - **趨勢預測卡**（如「CPU 持續上升」）— 有明顯趨勢的 metrics
   - **優化建議卡**（如「建議調整 log rotation」）— AI 根據 pattern 給的建議
   - **健康總覽卡**（如「過去 24 小時共 3 次異常」）— 總結性資訊
4. 使用者點擊卡片 → 展開詳細分析（AI 完整回覆 + 圖表 + 異常標註）

**輸入：** 當前 Dashboard 的 metrics（自動從 Prometheus 取得，無需使用者提供）
**處理：** AI 依據 dashboard 類型動態生成有意義的分析卡片
**輸出：** 可點擊的分析卡片組

**卡片觸發流程：**
```
使用者進入 Grafana Linux OS Dashboard
         │
         ▼
系統從 URL 識別 dashboard 類型（linux-os）
         │
         ▼
向 Prometheus 查詢該 dashboard 的核心 metrics
         │
         ▼
AI 分析 + 異常偵測 + 趨勢預測
         │
         ▼
生成 3-6 張情境感知卡片
         │
         ▼
使用者點擊感興趣的卡片 → 展開詳細分析
```

### 7.2 異常偵測（Anomaly Detection）

異常偵測使用兩種統計方法：**z-score** 和 **IQR**，兩者同時使用互補不足。

---

#### 7.2.1 z-score（標準分數）

**原理：** 衡量一個數值偏離平均值有多少個「標準差」

**公式：**
```
z-score = (該數值 - 平均值) / 標準差
```

**計算範例：**

假設過去 24 小時的 node_load1 數值為：

```
[1.2, 1.5, 1.3, 1.4, 1.1, 1.6, 1.4, 1.3, 1.5, 12.5, 1.4, 1.2, 1.3, 1.1, 1.4, 1.5, 1.3, 1.2, 1.4, 1.3, 1.5, 1.4, 1.3, 1.2]
```

**步驟 1：計算平均值**
```
平均值 = (1.2 + 1.5 + 1.3 + ... + 1.2) / 24 ≈ 1.7
```

**步驟 2：計算標準差**
```
標準差 = √(Σ(每個值 - 平均值)² / 24) ≈ 2.2
```

**步驟 3：計算 z-score**
```
第 10 個值（12.5）的 z-score = (12.5 - 1.7) / 2.2 ≈ 4.9
```

**步驟 4：判斷是否異常**
- |z-score| > 2.5 → 異常（偏離太多）
- |z-score| > 2.0 → 警示（值得注意）
- |z-score| < 2.0 → 正常

此例中 12.5 的 z-score = 4.9，遠超 2.5，**強烈異常**。

**視覺化：**

```
          正常範圍              警戒區          異常點
    ◄────────|────────|────────|────────|────────|────────►
           -2σ    均值    +2σ   +2.5σ    +3σ     +4.9σ
                                       ▲
                                       12.5 (z = 4.9)
```

**為什麼用 z-score：**
- 不需要 training data，適用於任何 metric
- 計算快速（毫秒級），適合即時分析
- 可解釋，異常點可以直接標示在圖上
- 缺點：對極端值敏感，數值分佈不是常態分佈時效果打折

---

#### 7.2.2 IQR（四分位距）

**原理：** 用資料的「中間範圍」當正常區間，超出這個範圍的就是異常

**概念：**
```
把所有數值從小到大排序，找到三個切割點：

Q1（第一四分位）──────── Q2（中位數）──────── Q3（第三四分位）
  25% 的資料                      50% 的資料           75% 的資料

IQR（四分位距）= Q3 - Q1

正常範圍：Q1 - 1.5*IQR ～ Q3 + 1.5*IQR

低於 Q1 - 1.5*IQR → 異常低
高於 Q3 + 1.5*IQR → 異常高
```

**計算範例（同一組 node_load1 資料）：**

假設排序後：
```
Q1 = 1.3（25% 位置）
Q2（中位數）= 1.4
Q3 = 1.5（75% 位置）
```

**步驟 1：計算 IQR**
```
IQR = Q3 - Q1 = 1.5 - 1.3 = 0.2
```

**步驟 2：計算正常範圍**
```
正常下限 = Q1 - 1.5*IQR = 1.3 - 1.5*0.2 = 1.0
正常上限 = Q3 + 1.5*IQR = 1.5 + 1.5*0.2 = 1.8
```

**步驟 3：檢查每個值**
- 12.5 > 1.8 → **異常高**
- 低於 1.0 的值 → **異常低**

**視覺化：**

```
    異常低          正常範圍                 異常高
  ◄─────────────|──────────────────────────────|─────────►
              1.0            1.4              1.8
             (Q1-1.5*IQR)   (Q2中位數)       (Q3+1.5*IQR)

     ▲ 低於 1.0                              ▲▲ 高於 1.8（12.5）
```

**為什麼兩種方法一起用：**

| 特性 | z-score | IQR |
|---|---|---|
| 適用分佈 | 常態分佈較好 | 不限分佈形態 |
| 對極端值 | 敏感（容易被拉高）| 穩健（少數極端值影響小）|
| 計算方式 | 需要標準差 | 只需要排序 |
| 適合情境 | 有明確上下界的 metrics | 無規律波動的 metrics |
| **同時使用** | 互補：z-score 抓到漸進偏離，IQR 抓到突發尖峰 |

**最終輸出格式：**
```
⚠️ 異常偵測結果
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[高] node_memory_MemAvailable_bytes
     時間：2026-05-04 03:00
     數值：128 MB（低於 Q1-1.5*IQR = 256 MB）
     依據：IQR 方法

[中] node_load1
     時間：2026-05-04 14:00
     數值：12.5（z-score = 4.9，超過警戒線 2.5）
     依據：z-score 方法
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 7.3 趨勢預測（Trend Prediction）

**方法：線性迴歸**

**原理：** 用過去 N 個時間點的數值，擬合一條直線，推估未來趨勢

**視覺化：**

```
CPU 使用率（%）
     │
100%│                              ▲ 預測點（未来 7 個）
     │                           ╱
 80%│                        ╱──●
     │                     ╱─●  ← 實際數值
 60%│                  ╱──●
     │             ╱──● ← 趨勢線（線性迴歸擬合）
 40%│        ╱──●
     │  ╱──●
 20%│─●
     │
     └──────────────────────────────▶ 時間
      過去                 現在       未來
```

**計算步驟：**

1. 收集過去 N 個時間點的數值（如過去 24 小時，每小時一個點）
2. 使用最小平方法（Least Squares）擬合直線：y = mx + b
3. 外推未來 7 個時間點
4. 標示「何時會突破警戒線」

**限制：** 只能抓到線性趨勢，複雜的非線性模式（如季節性波動）無法完美預測。這是合理的取捨：
- 計算代價極低（毫秒級）
- 對於「磁碟空間什麼時候滿」這類問題足夠準確
- Phase 2 可升級為 Prophet 等更複雜模型

## 8. Phase 2：知識庫增強

### 8.1 為什麼需要知識庫

沒有知識庫時，AI 只能根據「數字」分析。有了知識庫，AI 可以：
- 查詢「上次記憶體爆掉的原因是什麼」
- 引用「維運手冊寫的標準處理流程」
- 回答「這個錯誤碼代表什麼，該怎麼處理」

### 8.2 文件攝取流程

```
工程師上傳 PDF/MD/DOCX
         │
         ▼
┌───────────────────────────────────────┐
│  檔案解析 → 文字提取 → 語意分塊        │
│                                       │
│  PDF（PyMuPDF）                       │
│  Markdown（純文字）                    │
│  DOCX（python-docx）                  │
│                                       │
│  每 512 tokens 一個 chunk             │
│  chunk 之间 64 tokens 重疊            │
└──────────────────┬────────────────────┘
                   ▼
        ┌─────────────────────┐
        │  Embedding Model     │
        │  (將文字轉成向量)     │
        │  Ollama / OpenAI     │
        └──────────┬────────────┘
                   ▼
        ┌─────────────────────┐
        │  存入 pgvector       │
        │  (PostgreSQL 向量欄) │
        └─────────────────────┘
```

### 8.3 AI 如何使用知識庫

```
使用者：「記憶體快滿了怎麼辦？」

系統處理：
1. 向 Prometheus 查詢 memory metrics
2. 向 Vector DB 搜尋相關文件（top 5）
   → 「Linux 維運手冊 第3章：記憶體管理」
   → 「2024-11-03 事件報告：記憶體爆掉根因分析」
3. 組裝 Enhanced Prompt：
   「根據 metrics數據 + 以下文件，回答用戶問題」
4. AI 回覆：
   「根據《Linux維運手冊》第三章，建議步驟是...
    上次類似情況（2024-11-03）根因是 log rotation 失效，
    當時處理方式是...」
```

## 11. 風險與因應

| 風險 | 機率 | 影響 | 因應策略 |
|---|---|---|---|
| **AI 回答幻覺（hallucination）** | 中 | 高 | 降級為純統計分析（異常 + 趨勢），不回傳 AI 摘要 |
| **Ollama 模型品質不足** | 低 | 中 | 提供 `AI_PROVIDER=openai` 切換，雲端備援 |
| **Prometheus 資料延遲** | 低 | 中 | 回傳「資料可能不完整，請確認 Prometheus 狀態」|
| **Vector DB 搜尋結果不相關** | 中 | 中 | Phase 2 加入 category filter，使用者可以指定文件類型 |
| **檔案上傳安全問題** | 低 | 高 | Phase 2 上傳僅允許 PDF/MD/DOCX，限制檔案大小 50MB |

## 12. 名詞解釋

| 術語 | 說明 |
|---|---|
| **Prometheus** | 開源監控系統，收集並儲存 metrics（時間序列數據）|
| **PromQL** | Prometheus Query Language，用來查詢 metrics 的語言 |
| **Grafana** | 視覺化儀表板，支援多種數據來源（含 Prometheus）|
| **LLM / LLM** | 大型語言模型，如 GPT、Llama，用於自然語言處理 |
| **Ollama** | 本地部署 LLM 的工具，免費、無網路延遲、資料不外洩 |
| **RAG** | Retrieval-Augmented Generation，結合知識庫增強 AI 回答的技術 |
| **pgvector** | PostgreSQL 的向量搜尋擴充，功能是「找相似的內容」|
| **embedding** | 將文字轉成數字向量，讓電腦知道「哪些文字意思相近」|
| **z-score** | 標準分數，一種異常偵測方法。計算方式：z = (該數值 - 平均值) / 標準差。z > 2.5 表示數值偏離正常範圍，視為異常。適用於分佈接近常態的 metrics。 |
| **IQR** | Interquartile Range（四分位距），另一種異常偵測方法。將資料排序後，取 25%（Q1）、50%（Q2）、75%（Q3）三個點。IQR = Q3 - Q1。正常範圍為 Q1 - 1.5×IQR 至 Q3 + 1.5×IQR，超出者為異常。適合非正規分佈的資料，比 z-score 穩健。 |
| **Plotly** | 互動式圖表庫，支援 hover、zoom、export |
| **Chunk** | 文件分割後的小段落，便於 AI 處理和向量搜尋 |
